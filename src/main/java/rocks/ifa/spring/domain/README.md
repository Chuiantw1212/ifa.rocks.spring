# 模組指南：領域層 (Domain Layer)

本模組是整個系統的絕對核心，完整封裝了所有的核心業務知識、業務規則與業務狀態行為。本層級不關心任何底層的技術實現、資料庫結構或外部通訊協定，保持純粹的物件導向設計。

## 核心職責

* 業務領域知識封裝：專注於展現真實業務世界的邏輯與規則，確保業務不變性在任何交易過程中皆能獲得滿足。
* 狀態行為驅動：領域物件不僅包含資料，更包含與該資料相關的業務行為方法，拒絕貧血模型。
* 核心合約定義：定義系統核心所需的儲存庫介面與閘道介面，確立系統需要什麼能力，而將如何實現交給外層。

## 包含組件

* 聚合根 (Aggregate Roots)：聚合的邏輯邊界與唯一入口，負責協調內部的實體與實質物件，並對外提供統一的業務方法。
* 實體 (Entities)：具備生命週期與連續性概念、擁有唯一識別碼的核心領域物件。
* 實質物件 (Value Objects)：不具備識別碼、完全藉由屬性值來定義特徵且不可變的物件，如強型別識別碼或封裝金額與幣別的物件。
* 儲存庫介面 (Repository Interfaces)：定義核心領域資料的持久化與檢索合約，純粹使用領域物件作為參數與回傳值。
* 領域事件 (Domain Events)：表達過去發生且對業務系統產生顯著意義的事實，用於聚合內部或跨限界上下文的非同步解耦。

## 相依性約束與方向

* 允許依賴：本模組處於架構的最內層，絕對禁止依賴任何外部模組。其 pom.xml 中嚴格禁止出現 spring-web、spring-data-jpa、mybatis 或任何其他技術框架的相依性，僅可依賴原生 JDK。
* 被依賴性：本模組由應用層 (Application Layer) 與基礎設施層 (Infrastructure Layer) 依賴。

## 設計防線與最佳實踐

* 捍衛核心純粹性：嚴禁將持久化標註（如 JPA 的 @Entity 或 @Table）直接標示於本層的領域物件上，避免底層資料庫結構與業務邏輯發生緊密耦合。
* 不可變性原則：實質物件必須設計為不可變，一旦建立其內部狀態即不可修改，若需變更則產生全新物件替換。
* 細粒度聚合設計：保持聚合的輕量與高效，不同聚合之間應嚴格避免透過物件參照直接耦合，改以唯一識別碼進行關聯，以確保聚合能獨立載入與儲存。
