# 專案架構指南：COLA 5.0 四層架構約束

本專案架構嚴格遵循 COLA 5.0 規範，透過標準四層架構與多維度的專案結構，嚴格規範各層級的職責與相依性，以落實核心業務與技術實作的徹底解耦。

## 架構分層與設計約束

### 使用者介面層 (Adapter Layer)
* 核心職責：作為系統的流量入口，負責外部協定適配與參數解析。
* 包含組件：處理 HTTP 請求的 Controller、監聽 MQ 訊息的 Consumer 等。
* 設計約束：層內嚴禁包含任何業務邏輯。
* 允許的依賴方向：僅可依賴應用層與客戶端 DTO。

### 應用層 (Application Layer)
* 核心職責：扮演業務編排中樞。
* 包含組件：負責宣告交易邊界（@Transactional）、權限校驗、操作日誌。
* 運作機制：負責協調領域服務與基礎設施介面，每個功能通常對應一個專屬的 Executor。
* 允許的依賴方向：依賴領域層與基礎設施的抽象介面。

### 領域層 (Domain Layer)
* 核心職責：作為系統的絕對核心，完整封裝所有領域知識。
* 包含組件：包含聚合根 (Aggregate Root)、實體 (Entity)、實質物件 (Value Object) 與領域事件 (Domain Event)。
* 設計約束：保持純粹性，無任何技術框架依賴，僅依賴原生 JDK。

### 基礎設施層 (Infrastructure Layer)
* 核心職責：提供技術實現的插件化支援。
* 包含組件：對資料庫的存取實作 (Repository Impl)、對外 RPC 呼叫的實作 (Gateway Impl) 與設定載入。
* 允許的依賴方向：依賴領域層與各類第三方技術框架。

## 架構演進守則

全體開發團隊必須共同遵守此架構約束。當未來需要變更技術細節（例如將儲存層從關聯式資料庫遷移至 NoSQL）時，應確保僅需在基礎設施層抽換新的轉接器（Adapter），而完全不需要修改任何一行位於應用層或領域層的程式碼，以此展現六邊形架構應對技術變更時的韌性 [cite: 11]。
