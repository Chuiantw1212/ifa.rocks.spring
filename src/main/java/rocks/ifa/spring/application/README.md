# 模組指南：應用層 (Application Layer)

本模組扮演系統的業務編排中樞，負責協調領域層與基礎設施層的抽象介面，實現具體的業務案例。本層級關注的是業務流程的組合與系統層面的橫切關注點，而不包含核心的領域規則。

## 核心職責

* 業務流程編排：調度聚合根或領域服務，以循序且邏輯完整的方式完成特定的業務案例。
* 交易邊界控管：宣告交易邊界，確保整個業務案例的執行滿足原子性與一貫性。
* 橫切關注點處理：負責權限校驗、操作日誌記錄、審計追蹤等系統級功能。

## 包含組件

* 業務編排執行器 (Executors / Command Handlers)：每個具體的功能或命令通常對應一個專屬的 Executor，負責驅動單一業務情境的完整流轉。
* 應用服務 (Application Services)：協調領域對象與外部抽象介面的中介服務。
* 查詢服務 (Query Services)：在 CQRS 架構下，負責調度基礎設施層的查詢組件，將檢索結果封裝為表現層所需的唯讀模型。

## 相依性約束與方向

* 允許依賴：本模組依賴領域層 (Domain Layer) 以及基礎設施層定義的抽象介面。同時可調用公共合約與客戶端模組 (Common/Client) 的 DTO。
* 被依賴性：本模組由使用者介面層 (Adapter Layer) 調用。

## 設計防線與最佳實踐

* 嚴禁下沉業務規則：本層級僅負責流程的組合（例如：先從儲存庫撈取資料，再調用領域模型的方法，最後發送事件），絕對禁止包含任何核心業務邏輯。業務規則必須完整封裝於領域層中。
* 依賴抽象而非具體：本層在調用基礎設施功能（如發送郵件或存取資料庫）時，必須透過領域層或本層定義的抽象介面進行，具體的技術實現則由基礎設施層動態注入。
* 交易宣告守則：交易註解（@Transactional）應宣告於本層的業務入口處，確保整個應用案例在同一個交易上下文中安全執行，避免在領域模型內部進行交易控管。
